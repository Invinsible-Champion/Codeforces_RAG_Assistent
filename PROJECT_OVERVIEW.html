Section 1

System Architecture

When a user enters their Codeforces handle and clicks "Get Problem", the system executes a fully automated, five-step pipeline that bridges raw API data to a personalized coaching experience.

Design Decision
The system uses no intent routing. Unlike chatbot architectures that classify user messages into intents, this system has a dedicated UI tab for problem retrieval. The user's action (clicking "Get Problem") is the intent. This eliminates an entire LLM call and reduces latency.
Phase 1 — Foundation

Data Ingestion & Vector Storage

Before the AI can recommend anything, it needs a deep understanding of both the user and the available problem space. This phase builds the knowledge base.

Codeforces API Integration

The data pipeline (pipeline.py) hits the Codeforces problemset.problems endpoint to fetch every problem on the platform along with its solve statistics. For each problem, it extracts:

  • problem_id — Unique identifier (e.g., 1555A)
  • rating — Official difficulty rating (800–3500)
  • tags — Algorithmic categories (dp, graphs, math, etc.)
  • solved_count — Global number of successful submissions

PostgreSQL Storage

All problems are upserted into a PostgreSQL database via SQLAlchemy models. The pipeline then generates a natural-language description for each problem using template-based text generation. Example:

"The problem 'Watermelon' (ID: 4A) is a beginner-friendly competitive programming challenge with an official difficulty rating of 800. It focuses heavily on the following algorithmic concepts: math, brute force. With over 180,000 successful submissions, this is a highly popular problem, making it an excellent choice for practicing math, brute force logic."

ChromaDB Vector Store

These descriptions are embedded using SentenceTransformers (all-MiniLM-L6-v2) and stored in ChromaDB with HNSW cosine similarity indexing. This transforms the static problem database into a semantically searchable knowledge base.

Key Advantage
By embedding rich descriptions rather than raw tags, the semantic search can understand conceptual queries like "tree traversal with backtracking" even if no problem has that exact tag combination.
Phase 2 — Intelligence

Psychological Profiling

Instead of just looking at what a user solved, the system analyzes how they code. It translates raw JSON submission logs into a behavioral profile using 16 engineered features.

Data Acquisition

The ML Engine (ml_engine.py) makes two Codeforces API calls per user:

  • user.status — Complete submission history (every attempt, verdict, timestamp)
  • user.info — Current live rating

Submissions are parsed into a Pandas DataFrame sorted by problem_id and creation_time, forming a timeline of every coding decision the user has ever made.

Unsupervised Persona Clustering (GMM)

The 14 most discriminative features are StandardScaler-normalized and fed into a pre-trained Gaussian Mixture Model with 10 components. The GMM outputs:

  • A cluster assignment (0–9) representing the user's behavioral archetype
  • A probability vector quantifying classification confidence
  • A persona-specific system prompt loaded from persona_prompts.py

The Negative Filter

Finally, the engine compiles a strict set of every problem_id the user has ever submitted code for. This set is passed to the recommender to ensure they never receive a duplicate.

Why GMM over K-Means?
GMM provides soft clustering — it returns probabilities across all 10 archetypes, not a hard label. This means a user who is 60% "Methodical Optimizer" and 30% "Solid Specialist" gets a nuanced system prompt, and the confidence score is exposed in the Analytics dashboard.
Phase 3 — Core Engine

The Two-Stage Recommender

This is the absolute core of the architecture. When a user requests a problem, the system executes a highly optimized retrieval and ranking pipeline in chat_engine.py.

Stage 1: LLM Query Builder + Broad Retrieval

The engine calculates a target rating (average of the user's historical solved rating and current live rating) and identifies their weakest algorithmic domain from the ML profile.

A lightweight Gemini 2.5 Flash call receives the user profile and outputs a strict JSON search query:

{ "search_query": "dynamic programming state transition optimization", "min_rating": 1100, "max_rating": 1500 }

ChromaDB executes this query with metadata filtering ($gte/$lte on rating) and over-fetches 50 candidate problems to give the ranker maximum choice.

Graceful Fallback
If the Gemini API is unavailable (503/429), the system automatically falls back to the user's weakest domain from the ML profile as the search query — no user-facing error.

Stage 2: Negative Pruning

The 50 candidates are cross-referenced against the user's attempted_problems set compiled in Phase 2. Any problem the user has ever submitted code for — even failed attempts — is instantly dropped. This guarantees freshness.

Stage 3: Dynamic Feature Assembly

For each surviving candidate, the engine dynamically constructs a 20-feature vector in real-time:

#Feature GroupFeatures (Count)
1–8User Metricsaccuracy, optimization_struggle, avg_solved_rating, abandonment_rate, one_shot_rate, tilt_speed, recent_win_rate, persistence_index
9–11Problem Statsproblem_rating, global_solve_rate, avg_attempts_per_user
12Cross Featurerating_delta = problem_rating − user_avg_solved
13–20Tag Synergyis_dp × dp_pref, is_math × math_pref, is_graph × graph_pref, is_brute × brute_pref, is_greedy × greedy_pref, is_binary × binary_pref, is_cons × cons_pref, is_datastruct × datastruct_pref
Tag Synergy Explained
Each synergy feature multiplies a binary "problem has this tag" flag (0 or 1) by the user's continuous proficiency score in that domain (0.0–1.0). A high synergy score means the problem aligns with the user's existing strengths; a low score means it targets a weakness — both are useful signals for the ranker.

Stage 4: XGBoost V2 Re-Ranking

The assembled DataFrame is fed into a pre-trained XGBoost Classifier (xgboost_ranker_v2.json). The model predicts the probability that the user will successfully solve each problem (P(y=1)).

Problems are sorted by this confidence score in descending order. The highest-ranked problem is selected as the final recommendation.

# Predict probability of success for all candidates probabilities = ranker_model.predict_proba(df_scoring)[:, 1] # Sort by confidence and select the winner ranked = sorted(candidates, key=lambda x: x["xgb_score"], reverse=True) winning_problem = ranked[0]
Phase 4 — The Voice

Pedagogical Presentation

Data science is only as good as its presentation. The final step bridges the gap between cold math and human mentorship.

Contextual LLM Injection

The winning problem's full context — title, rating, tags, and raw description text — is injected into a final Gemini prompt alongside the user's persona-specific system prompt from Phase 2.

presentation_prompt = f""" {system_prompt} You are presenting this freshly retrieved Codeforces problem to the user: {context_str} INSTRUCTION: Introduce this problem perfectly in your persona's character. Explain exactly WHY you chose this specific problem based on their ML profile weaknesses. Do not give them the solution. Be concise and format nicely with markdown. """

Persona Execution

The LLM doesn't just present the problem — it coaches. Using the persona identified by the GMM, it:

  • Introduces the problem with appropriate enthusiasm (or strictness)
  • Explains why this specific problem was selected based on their weaknesses
  • Provides strategic encouragement tailored to their learning style
  • Never reveals the solution — it's a coach, not a tutor

Graceful Degradation

If the Gemini API fails during presentation, the system falls back to a raw markdown rendering of the problem details directly from the database — the user always gets their recommendation.

Section 7

Persona Catalog

The GMM clusters users into one of 10 behavioral archetypes. Each archetype has a unique coaching personality, strategy, and set of problem selection biases.

IDNameRatingKey TraitCoaching Strategy
0 The Cautious Beginner ~1090 High accuracy, weak DS Gently introduce O(N log N) and data structures
1 The Impatient Novice ~1170 Greedy-reliant, 35% quit Enforce rigorous mathematical proof before coding
2 The Persistent Grinder ~970 Great grit, lacks theory Praise + deep conceptual algorithm explanations
3 The Fickle Advanced ~1560 Strong skills, 45% abandon Break problems into micro-tasks, build resilience
4 The Stubborn Graph Hacker ~1665 Graph expert, TLE issues Enforce Big-O optimization before debugging
5 The Comfort Zone Camper ~880 Pads stats with easy Force progressive overload, refuse easier problems
6 The Methodical Optimizer ~1190 Disciplined, strong BS Bridge to "Binary Search on Answer" paradigms
7 The Solid Specialist ~1420 DP & Graph focused Peer-level: focus on DP state transitions
8 The Advanced Precisionist ~1560 One-shot precision Multi-tag challenges, space complexity optimization
9 The Frustrated Flounderer ~980 Guesses often, tilts fast De-escalate, rebuild confidence with easy problems
Why Persona-Driven Coaching Matters
A user classified as "The Frustrated Flounderer" needs empathy and easy wins. Giving them the same 1600-rated DP problem as "The Advanced Precisionist" would be counterproductive. The persona system ensures the same pipeline produces radically different outputs based on individual psychology.
